Get to Grips With Windows Server

In this first part, you will learn what Windows Server is! First, you will learn to distinguish between the different editions of this operating system and discover how to install it. Before we start, remember that Windows is a standalone Microsoft brand. Among other things, it has its own logo:

Understand the Context

Microsoft Windows Server is Microsoft’s server operating system. In a sense, it is an enhanced version of the Windows system you are probably familiar with, which comes installed by default on most consumer computers!

Why distinguish between a user operating system and a server operating system?

Because a server operating system must meet specific requirements:

  • high stability and availability

  • remote administration

  • enhanced security

  • automation (PowerShell, scripts, etc.)

Windows Server is therefore not simply an improved version of Windows client. It is a system designed for services and infrastructure.

To do this, Microsoft chose to segment its different systems into several major families. In 1985, Microsoft released Windows 1.0. The name refers to the windowing system that simplified computer use, which at the time was done entirely through a shell, a command-line interface. At Microsoft, this system was called DOS, or Disk Operating System.

This window-based system quickly proved very practical and evolved fast. During the 1990s, the need to compete with the server systems of the time pushed Microsoft to design a system focused on services rather than office productivity. With the NT versions, Windows Server was on its way.

The popularity of this new version of Windows, Windows NT, in the business world pushed Microsoft to keep investing in this area. In 2003, the first version of Windows Server finally arrived: Windows Server 2003. This version marked a turning point in Microsoft’s expansion into enterprise environments. It was massively adopted by businesses. New versions of this system then followed regularly: 2008, 2012, 2016, 2019, 2022, and 2025.

In 2026, the 2019 and 2022 versions are still very widely used. Windows Server 2025 is starting to be adopted, especially in new projects and secure environments. Earlier versions, such as 2016, 2012, 2008, and 2003, are now obsolete and are often maintained only for legacy business applications.
The fundamentals of installation, configuration, role and feature administration, and PowerShell use remain broadly similar across 2019, 2022, and 2025.

However, security, cloud integration, automation, and modern administration have evolved significantly.

In this course, we will use Windows Server 2025 while highlighting what remains consistent with previous versions.

Distinguish Between the Different Editions of Windows Server

Windows Server 2025 comes in several editions to meet different needs depending on the size of the organization, the level of virtualization, and the infrastructure deployment model, whether on-premises, in the cloud, or hybrid. Choosing the edition is an important decision. It determines the available features, virtualization capabilities, licensing model, and, most importantly, licensing cost, because the editions are not sold at the same price.

  • Windows Server 2025 Essentials
    This edition is designed for small businesses and organizations with up to 25 users and 50 devices. This license lets you run, at a lower cost, all the services you will master over the next few chapters. Just purchase the hardware, such as a server and workstations with a Windows Professional license, and you are all set!

  • Windows Server 2025 Standard
    This edition is designed for any organization that does not have major virtualization needs, or that has low density, meaning few servers. Its pricing is based on the number of cores on the physical server. You will need to purchase Client Access Licenses (CALs) in addition to purchasing a workstation with a Windows Pro license.

  • Windows Server 2025 Datacenter
    This high-end edition is designed for highly virtualized environments. With this edition, you can create as many Windows Server virtual machines as you want. Its pricing is also based on the number of cores and also requires a Client Access License (CAL) for each client computer, in addition to a Windows Pro license.

  • Windows Server 2025 Datacenter: Azure Edition
    This edition is designed for hybrid environments and certain deployments related to Azure. It differs from the "classic" Standard/Datacenter editions through its cloud- and hybrid-oriented positioning and features specific to this model.

As you can see, you need to define your needs carefully to choose the right edition of Windows Server 2025.

Choose a Recent Version of Windows Server

Wait, before moving on, here is some additional information that may interest you.
When you get oriented in an existing company, you will find other versions of Windows Server.

Why?

Simply because the company you join has an IT history in addition to its own business history. Updating an existing server to a new version is a complex step because it requires servers to be shut down. With data security at the center of everyone’s concerns, shutting down services is becoming increasingly complicated. Doing so would make data unavailable by making services unavailable. What does that have to do with security? One of the key criteria for data security is availability!

So this raises a question:

Why work on moving to a more recent version of Windows Server?

Compared with previous versions, Windows Server 2025 marks a significant evolution, especially in the following areas:

  • Windows Server Hotpatching: Allows certain security updates to be installed without restarting the server, reducing service interruptions.

  • Next-generation Active Directory and SMB: Strengthens identity and file access security, especially through better encryption and the gradual retirement of obsolete protocols.

  • Hyper-V, GPU, and artificial intelligence (AI): Improves virtualization with better GPU support, making intensive workloads and AI-related use cases easier to manage.

  • Connectivity and hardware: Simplifies the use of recent hardware, including Wi-Fi enabled by default when the hardware is compatible.

As you can see, even though updating an operating system is complex, new features can provide a real advantage and help you increase the security of your IT system, and more generally your information system!

Prepare Your System for Networking

In this chapter, you will prepare your Windows Server for networking. Why dedicate a whole chapter to this? Because this is where most errors happen. You need to understand exactly what you are doing and what you can do, because Microsoft has automated many settings.

If you do not prepare your system, you will create serious vulnerabilities that could compromise your network security. Even worse, they could provide unwanted access to your server and therefore to the data it hosts or manages!

This first step on your virtual machine is the equivalent of plugging a network cable into a physical server. Name a Server

If you have already taken the course on Active Directory, you know that naming conventions are not something to take lightly. Naming a device correctly—in this case, your server—makes it possible to track its life cycle.

It is also important not to name the server based on its role or public name. Instead, choose a consistent, logical, and simple naming convention. In any case, you can assign other names to the server through DNS!

For example, to name your servers, you could use a code such as SRVADNYC01 for a server (SRV) with the AD DS role, located in New York City (NYC), and which is the first server of this type (01).

This makes it easy to find all servers hosted in New York City by searching for "*NYC*", or to find all Active Directory servers by searching for "*AD*". I will let you think through a convention that makes sense!

To rename the server, simply click its name in the dashboard:

You will need to restart for the name change to take effect. Next, you need to decide what your server’s IP address will be on your network.

Configure Your Server’s TCP/IP Layer

This step depends on your network, any subnets, and the addressing strategy you have implemented!

Here, your server is behind a router that performs Network Address Translation. As long as it is operating in this mode, it must be configured with an address from this VirtualBox-specific network.

Still in the dashboard, if you click "IPv4 address assigned by DHCP, IPv6 enabled", you will access the configuration for your network interfaces. If you are familiar with Windows as a desktop operating system, you will feel at home. Right-click the active network adapter, then click "Properties":

You will notice that Microsoft enables many services on an interface by default:

  • Client for Microsoft Networks;

  • File and Printer Sharing for Microsoft Networks;

  • QoS Packet Scheduler;

  • Internet Protocol Version 4 (TCP/IPv4);

  • Microsoft LLDP Protocol Driver;

  • Internet Protocol Version 6 (TCP/IPv6);

  • Link-Layer Topology Discovery Responder;

  • Link-Layer Topology Discovery Mapper I/O Driver.

IPv6 is enabled by default. If your network does not need it, I recommend disabling it.

Be careful, though: some Microsoft roles use this protocol. It may be wiser to study the role in detail—which you will do in this course—to understand how it works, and then decide whether keeping IPv6 is relevant.

You are not quite finished yet. Select the properties for "Internet Protocol Version 4 (TCP/IPv4)", then click "Advanced":

The first tab lets you configure all IP settings: address, subnet mask, aliases, and gateway(s).

The second tab, DNS, lets you configure everything related to name management. A first area lets you configure the DNS server addresses:

For Linux users, this is the equivalent of the "resolv.conf" file. In the "Suffixes" section, you can specify how DNS queries will be sent to the server, either directly or by adding a suffix.

Note that you can also manage the registration of the machine name and its address in your internal DNS. No need to worry: you will explore the DNS role in detail in the following chapters.

Finally, there is one last tab: WINS. This is a protocol similar to DNS, but specific to Microsoft and Windows (Windows Internet Naming Service). It was widely used, and even mandatory in some cases, before the 2000s and the widespread adoption of Active Directory (which uses DNS). It made it possible to find an IP address from a NetBIOS name:

What is NetBIOS? And it is enabled by default too?!

NetBIOS is not a protocol! It is a naming system and software interface that establishes sessions between different computers on a network. This communication program is gradually being replaced by DNS. However, Microsoft still uses it today over TCP/IP, particularly for compatibility with Windows NT and XP systems.

You may have noticed the "Enable LMHOSTS lookup" checkbox. LMHOSTS is to NetBIOS what the HOSTS file is to DNS. It lets you create a manual association between a NetBIOS name and an IP address! Again, it is worth disabling if you no longer need NetBIOS.

Once you have finished these IP configurations, and TCP/IP configuration more generally, your server is ready to be networked.

Unless there is one setting left to configure...

Configure Windows Defender Firewall

Windows includes a firewall. Like Linux and iptables, it lets you lock down incoming and outgoing network traffic on your server! This is an essential extra layer of security today. Why leave port 80 open on a server that does not have the Web Server role?

Once again, go to your Local Server dashboard:

As you can see, I chose to configure a first DHCP server in New York City: SRVDHCPNYC01. I now only have IPv4 with an address, a subnet mask, a gateway, and a DNS server (no NetBIOS and no other Microsoft network services). My server is operational.

To configure the firewall, you have several options:

  • Click the line that says "Windows Firewall: Private On";

  • Enter "Defender" (the name of the Windows firewall) in the search bar.

The first method opens a simplified firewall configuration window. Personally, I use the second method because it opens a window with more configuration settings.

That is a pretty good default configuration: it blocks all connections to applications that are not on a list! For peace of mind, click "Change notification settings" and select the options "Block all connections and notify me when access occurs":

This way, your server can initiate outbound network connections, which is useful for configuring roles, features, and services, but it will not be accessible from the network!

Note that this firewall step can be done first. Being behind VirtualBox’s NAT mechanism gives you that flexibility.

You may have noticed that Windows automatically classifies your networks into two connection profiles: private connections and public connections. This allows you to create two configuration profiles on your firewall: one profile with very restrictive rules when you use a public network, and a slightly more flexible profile when you use a private network.

That’s it: your server is ready to be connected to a network.

Understand Roles, Features, and Services

Roles, features, services... What do these terms mean?

A Windows server in a company will have one or more roles. These roles might include providing or hosting files, managing a directory service (AD DS), managing certificates (AD CS), or managing network configuration (DHCP).

Each of these roles relies on services to run. These are the equivalent of daemons in Linux/Unix. For example, the svchost.exe service with the "-k DHCPServer" option is required for the DHCP role to work.

Finally, to be effective, a role may need features. The Web Server role that hosts your web pages may need a .NET feature if you need to develop your web application using a dynamic language.

Set up server monitoring

In this final chapter of Part 1, let’s work on a topic that is often forgotten or handled hastily: monitoring. Good monitoring helps you avoid many availability, inventory, and broader security issues!

Get familiar with automatic Windows Server monitoring

Microsoft fully embraced the graphical interface approach, especially with Server Manager. You don’t need to look any further to monitor your server! And that’s it, the course is over, thank you... Just kidding. Let’s dig deeper.

Note: Some screenshots and the video in this chapter may show the French Windows Server interface. In en-US Windows Server, use the UI labels referenced in the instructions, such as Dashboard, Local Server, Tasks, Start BPA Scan, Configure Performance Alerts, and Start Performance Counters.

Let’s analyze what this automated monitoring includes. Go to the dashboard, which you now know well:

Remember: on the left, you have the quick access menu. In the center is the dashboard itself, with a Quick Start area and monitoring items. In the upper right, you’ll find the menu with notifications, shown as a flag icon.

First, the left-hand menu. It lets you quickly access a server, a role, or your server groups:

Let’s stay in the Local Server section!

I won’t go back over the first area, where you can find your server’s basic settings, such as network, name, and update status. Below that, you’ll find events from your server’s different event logs.

Microsoft has implemented a system similar to “journalctl” on Linux. All events are recorded in these logs, and the dashboard displays them all here. Install a DHCP Server

In this second Part, you’ll get hands-on with different Microsoft roles and install them on your server. You’ll start with the DHCP Server role, which, as you probably already know, automates device configuration within your network.

Review the DHCP fundamentals

Here’s a quick refresher on how DHCP works. Dynamic Host Configuration Protocol is, as its name suggests, a dynamic (automatic) configuration protocol for machines. This removes the risk of mistakes when configuring the network settings for your devices, making your network more reliable.

Microsoft implements RFC2131 within the DHCP role.
In short, the DHCP server listens on UDP port 67.
If you’d like to learn more, feel free to read this section of the course on how TCP/IP networks work.

Install the DHCP Server role

Click the notification, then click "Complete DHCP configuration". You should arrive at this screen:

The wizard shows you the different steps required to configure this role. The first step is creating groups for administration delegation. This allows you to create a user whose only rights are to administer this role.

You can then close this wizard. Now, use the DHCP Server management console, which lets you administer and, most importantly, configure DHCP. To do this, go to the DHCP section, right-click, and select the DHCP console:

You’ll find all the options for configuring this role:

The window is divided into three areas: on the left, your DHCP servers; in the middle, the configuration options; and on the right, the available actions based on the context. In particular, this is where you’ll find action shortcuts to the menus that appear when you right-click.

Manage the DHCP service

As I mentioned, a role uses features that rely on services. The first thing you can do with this console is stop, start, or restart the service associated with the DHCP role. By default, the DHCP role can work with IPv4 and IPv6. In this course, I’ll only cover IPv4 because IPv6 adoption is still lagging. But don’t worry: you’ll probably need to work with IPv6 very soon.

You therefore have the following settings available:

  • server options;

  • policies;

  • filters.

By selecting IPv4 and right-clicking this setting, you can create address scopes to distribute with your server.

A scope is a range of IP addresses assigned to computers requesting a dynamic IP address.

You can then define options specific to each scope, as well as global options: these are the server options that apply to all scopes. Policies let you define conditions for providing options or addresses based on specific criteria. For example, you could specify a different default gateway for your Windows 2000 devices or, more plausibly, provide specific configuration options to devices with MAC addresses that appear in a list.

Finally, you can set up exclusions or allow rules through filters:

I recommend using both: record the MAC addresses of all your devices here, and only provide an IP address if the MAC address is in the list. You can also combine this with policies by specifying known client identifiers. That way, you control which devices can access your network.

Now that you understand how the DHCP console works, let’s create your first scope. To do this, right-click IPv4 under the server name, then click "New Scope":

Once again, Microsoft lets you configure this through a wizard.

Fill in the different fields however you want—or rather, according to your needs. In general, you’ll configure the address range (start and end addresses), the gateway, and the DNS server. You could also add options for IP phones, such as the telephony server.

For simplicity, start by choosing a scope that belongs to your IP network. In our case, the server was configured with the address 10.0.2.15/8, so we’ll ask the DHCP server to distribute, for example, addresses from 10.0.3.1 to 10.0.3.254.

The next screen lets you enter exclusions or delay server responses, which is useful when you have multiple DHCP servers on the same subnet:

Finally, the next screen lets you define the lease duration.

This duration is often left at the default value. Think carefully about it: a known cyberattack involves saturating DHCP servers by requesting large numbers of leases. If your lease duration is too long and you’re hit by this type of attack, you could bring automatic network configuration to a halt.

Be careful not to go too far in the opposite direction with a duration that is too short. That could congest your network by making your devices request lease renewals too often.

Install a DNS Server

In this chapter, you’ll install the DNS Server role on a server. This role is essential for many other roles, including Active Directory, the most commonly used one. It lets you map a fully qualified domain name to an IP address. In fact, the web as we know it today is based on this protocol.

Review DNS fundamentals

DNS stands for Domain Name System. It is a protocol that maps a name to an IP address.

A client, often a web browser, sends a request to find the IP address of the web server that matches the address you entered.

For example, if you go to www.example.com, your browser needs to ask for the IP address of the server named www in the DNS zone example.com.

To do that, it sends a query to the DNS server configured on your workstation and makes a type A request for www.example.com. If your DNS server does not know the answer, it then turns to another DNS server. This is often one of the root servers that manages records for the “.” zone.

This special zone references the servers for zones such as “.com,” “.fr,” and, more generally, “.domain name extension.”

Your server will then query the name server for the “.com” zone to find the DNS server for that zone. The process starts again until it finds the example.com zone, which then returns the A record for the www field in its zone. You should get the IP address 91.198.174.232.

Before moving on, if you are using a virtual machine, you’ll need to make a few changes to this VM.

As you can see in the previous diagram, your DNS server must be able to:

  • Receive queries from machines on your private network.

  • Send queries to public DNS servers on the internet.

It must therefore be connected to both your company’s private network and the internet, using a network card with a fixed IP address. The IP address of a DNS server, just like that of a DHCP server, must always be static so client machines can find it easily.

To get this configuration in VirtualBox—private network access, internet access, and a fixed IP address—you need to change the connection mode in VirtualBox.
To do this, shut down your VM and go to the VirtualBox settings:

Then click the “Create virtual adapter” tab, followed by “NAT Networks.”

You can now confirm the settings. All that’s left is to associate your Windows server’s network card with this new NAT network named NATNetwork.

To do this, go to your VM’s hardware configuration and select this new network mode:

Manage the DNS service

Just like DHCP Manager, there is a DNS Manager. This console, dedicated to administering the DNS role, lets you create the different zones required for DNS to work.

To access it, right-click your server, then select “DNS Manager”:

Name management is an area that is often taken too lightly. Remember that 100% of internet-bound requests go through DNS.

Before creating your first forward lookup zone, you need to understand how a DNS server works. For each client query, the response is cached locally. After your server retrieves the IP address for www.example.com, this cache lets it respond more quickly without launching another recursive query to the root servers and the server for the “example.com” zone. This cache must be managed.

Why does this cache need to be managed?

Quite simply, to avoid keeping the association between www.example.com and the IP address 91.198.174.232 in memory indefinitely.

That’s right. If the administrator of this server decides to change its IP address, it would be a shame if you could no longer access www.example.com.

Be careful to set the cache to a value that is neither too low nor too high.

By default, this value is one day, or 24 hours, for positive responses, where at least one IP address exists for a fully qualified domain name, and 15 minutes for negative responses. To display this information, open PowerShell and type the command Get-DnsServerCache:

We’ll return to using PowerShell in the last part of this course. Use Microsoft’s online help to modify these values.

Finally, make sure the DNS role is configured correctly. In DNS Manager, right-click the server name and choose “Properties”:

The first tab lets you define which interface your server will use to receive and accept DNS queries. In our case, we only have one interface, but it is very common for company servers to have several network interfaces.

So which interface should the DNS service listen on?

It all depends on your network. By default, it will listen for DNS queries on all interfaces, which is not a best practice. For example, if my server had two interfaces—one to communicate with the internet and another to communicate with the company’s private network—I would need to tell it to listen only for queries coming from the private network. That is where DNS queries will come from, so there is no reason to listen on the other interface too.

In our case, select only the fixed IP address you configured in the first chapter.

Go to the Root Hints tab. Check that your server knows the root servers. These are the servers that will be contacted to identify a response for www.example.com, for example.

You might be wondering: how do you test your DNS server? With these servers, shouldn’t it be able to answer any request?

On your server, open a command prompt, or shell. Type the command nslookup - 10.0.2.15, where 10.0.2.15 is the IP address of your server’s network interface. This opens an interactive DNS client connected to your server! Test www.example.com:

nslookup - 10.0.2.15

Serveur par défaut : UnKnown
> www.exemple.com

Serveur : UnKnown
Réponse ne faisant pas autorité :
Nom : www.exemple.com
Addresses : 2606:2800:220:1:248:1893:25c8:1946
            107.180.40.145

Because your server does not manage the example.com zone, it recursively queries the root servers, then the DNS server for the example.com zone. The response your server provides is not authoritative because your server got it from another server. 😅

How do you get authoritative responses?

By having a DNS zone. That works out well, because that’s exactly what you’re going to set up!

Set up your first forward lookup zone

A forward lookup zone lets you map a name to an IP address. That is much easier for us humans to remember, especially because an IP address can change over time, for example when you change internet service providers.

The first step is to choose a domain name. I suggest using a private zone.

Consider the following case: after the DHCP configuration, Gift Corp. management asks you to find a way to name the different devices and services on the network. The director is tired of typing the IP address 10.0.2.10 to access the intranet. You will therefore create a forward lookup zone for the domain “gift.example” and place an A record in it that points intranet.gift.example to 10.0.2.10. This way, your director can simply type https://intranet.gift.example instead of the IP address. Mission accomplished!

To do this, you have yet another wizard. Click Action at the top of your screen, then select “New Zone...”.

After the welcome screen, you should see the following screen:

The first choice is perfect; it’s what you want to do. Confirm this choice by clicking “Next” until you reach the zone name page. In this case, it is “gift.example”. Then comes the creation of the zone file and its location on your server. By default, the file will be named gift.example.dns. Next comes the question of dynamic updates.

Handle this option with caution because it allows a client to update records. If it is misconfigured, a malicious user could change your records to send them to the IP address of a server they manage, which could lead to a data leakage campaign. Be careful!

Your zone has been created. Now view its contents by clicking “Finish,” then expanding “Forward Lookup Zones”:

Your zone contains only two records, which identify the authoritative server, or SOA, and the name server, or NS. It would be useful to create the intranet record requested by management. To do this, right-click in the right-hand window, or on the zone name, and select “New Host (A or AAAA)”. A records are for IPv4, and AAAA records are for IPv6. Enter the host name within the zone, which is “intranet,” and the associated IP address:

The 10.0.2.10 server does not exist, but let’s assume it is the one hosting the intranet website.

Here, you can create a PTR, which you’ll see in the next section in a few minutes. Do not check this box, and confirm by clicking “Add Host”. Notice the non-editable field for the fully qualified domain name, or fully qualified domain name – FQDN. It includes a dot at the end that represents the root zone, followed by the reserved top-level domain “example” and then the domain “gift”. The fully qualified name for the intranet is therefore “intranet.gift.example.”

To verify that your record was created correctly, reopen a command prompt and type the command nslookup intranet.gift.example 10.0.2.15 to ask server 10.0.2.15 what IP address the host “intranet.gift.example” has:

>nslookup intranet.gift.sa 10.0.2.15
Server : UnKnown
Address: 10.0.2.15

Nom : intranet.gift.sa
Address : 10.0.2.10

There you go—you now know how to create A records in a forward lookup zone! You can name all your devices or servers and stop using IP addresses.

Discover other record types

Before moving on to the reverse lookup zone, let’s look at a few additional points. DNS responds to a client query. Type A lets you request an IP address from a name, but many other types are available, such as NS, which lets you identify the name server. In Windows, you can run queries for different types using the set type=XXX option, where XXX is the requested type.

Another type is SOA, which stands for Start of Authority and identifies which server is authoritative for a zone.

>nslookup - 10.0.2.15
Serveur par défaut : UnKnown
Address: 10.0.2.15

>set type=SOA
>gift.sa
Serveur : UnKnown
Address: 10.0.2.15

gift.sa
 primary name server = srvdhcppar01
 responsible mail addr = hostmaster
 serial = 2
 refresh = 900 (15 mins)
 retry = 600 (10 mins)
 expire = 86400 (1 day)
 default TTL = 3600 (1 hour)

>set type=NS
>gift.sa
Serveur : UnKnown
Address: 10.0.2.15

gift.sa nameserver = srvdhcppar01

Another type is CNAME, which lets you associate one name with another name.

This can be useful when you want to give a server a name that points to the service name. For example, it can be useful to have a different name for administering the intranet, but typing “intranet.gift.example” when administering this service can be lengthy. A CNAME “int.gift.example” pointing to intranet.gift.example solves this problem:

>set type=CNAME
>int.gift.sa
Serveur : UnKnown
Address: 10.0.2.15

int.gift.sa canonical name = intranet

There are many record types. The best known are NS, SOA, A, AAAA, CNAME, TXT, and MX, or Mail eXchange for mail servers. The method to implement them is the same, regardless of the type.

Install a File Server

One of the most common roles for a Windows server is the file server. With this role, you can securely make files available on your network. You can manage access permissions such as read, write, and modify. Providing this type of role on a network lets you centralize file storage, which makes backup and restoration easier and allows several people to work together on the same file.

Review file sharing fundamentals

For a file to be accessible on your network, you need to use a communication protocol designed for file sharing. On Linux, this is usually NFS (Network File System). On Windows, you can use this protocol, but it is not as tightly integrated into the system as the SMB protocol, also known as CIFS. SMB stands for Server Message Block, and CIFS stands for Common Internet File System.

This protocol includes a native client built into Microsoft Windows and a server that you will install and configure. It relies on NTFS to manage access permissions, and shares are accessible through a Universal Naming Convention (UNC) path such as \\serveur\partage.

If SMB reminds you of SAMBA, which is often used on Linux, that is completely normal. SAMBA is an open-source implementation of Microsoft’s proprietary SMB protocol. You can therefore share your files via SMB and access them from your Linux clients.

With that covered, let’s dive in!

Install the File Server role

To do this, shut down your virtual machine and go to the VirtualBox menu under "Machine," then select "Settings." You may have noticed that Windows asks you to provide a reason for shutting down your server:

When creating the share, the New Share Wizard gives you five options:

  • SMB Share - Quick: the simplest option, which provides a share on your network via SMB.

  • SMB Share - Advanced: goes further than the previous option by managing quotas and advanced permissions.

  • SMB Share - Applications: used for Hyper-V, databases, and other servers.

  • NFS Share - Quick: identical to the simple SMB option, but via NFS, with better Linux compatibility.

Install a Network Access Server

In this final chapter, you’ll implement a role that is too often overlooked: network access control. Microsoft provides this role so you can control which devices can access your network resources.

Imagine that your management team asks you to secure access to a shared resource over Wi-Fi, which is vulnerable by design. With this role, you can control your Wi-Fi and provide it to your management team in a managed way!

Install Network Policy and Access Services

The network access server role is installed through Network Policy and Access Services.

Known as NPS (Network Policy Server), these services provide access control for your network. This server acts as a RADIUS server (Remote Authentication Dial-in User Service), a protocol used to verify a client’s identity and permissions and provide a service, such as network access, if the client has those permissions.

NPS can also act as a RADIUS proxy, but we’ll focus on its primary function: the RADIUS server.

Install this role. In most cases, you shouldn’t need to restart your server. Select your server. In this example, I named my server SRVNACNYC01: it’s an SRV server for NAC (Network Access Control ), and it’s the first (01) server located at the New York site (NYC). Right-click the row for your server and select "NPS Server":

Here’s an additional note about Windows Server editions and RADIUS. The Standard edition of Windows Server can manage "only" 50 RADIUS clients, whereas the Datacenter edition does not restrict the number of clients. This can help you refine your criteria when choosing a Windows Server edition and size your network accurately.

Before you start implementing network access control, you need to understand the terminology and operation of RADIUS. These concepts are common to all access protocols.

Review the Fundamentals of Access Control

First, you need to identify the clients. You’ll need a client that supports the RADIUS protocol. The advantage here is using Microsoft technologies: you’ll find native clients for each server role.

That is the case here: a Windows 10 Pro workstation can query a RADIUS server. However, there is one caveat: the client workstation will not be the RADIUS client. It will not directly ask the access server whether it can access the network. To do that, it must rely on a third party that will query the RADIUS server.

Here is a workflow diagram for Microsoft’s network access role.

The client workstation will therefore be a supplicant in RADIUS terminology. The RADIUS client is the device that provides network access. It can be a switch, a wireless access point, or a remote access or VPN server.
