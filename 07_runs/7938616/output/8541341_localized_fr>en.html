Understand where PowerShell comes from

In this chapter, let's take a quick trip back to an earlier era of computing.

In the 1980s, the personal computing market was already dominated by Microsoft and its MS-DOS operating system.

MS-DOS was the operating system, or OS (for Operating System), installed by default on PCs. It was controlled through commands entered by the user.

That looks a lot like the Windows Command Prompt!

Eh oui ! L'command prompt Windows, ou cmd.exe. In fact, the Windows Command Prompt, or cmd.exe, is an emulation of MS-DOS.

Back in the MS-DOS era, the command prompt executable was command.com.

MS-DOS already had its own scripting language: Batch.

If you are interested in Batch syntax or want to learn more about this scripting language, the documentation is available on this Microsoft Learn page.

Since Windows moved away from MS-DOS and adopted the NT kernel starting with Windows XP in 2001, command.com has been replaced by cmd.exe.

Okay, so where does PowerShell fit into all of this?

We're getting there! In order to move forward, Microsoft decided to create another scripting language to replace Batch: MSH, for Microsoft command SHell, which later became Microsoft PowerShell.

PowerShell has been included directly in Windows since Windows 7.

Today, the built-in version of PowerShell in Windows (Server 2022 or Windows 11) is version 5.1. There is also a separate installable version called PowerShell Core. PS Core is currently on version 7.2 LTS, which stands for "Long-Term Support".

Chain your cmdlets

Imagine you want to retrieve all the items in a path that are directories.

How would you go about that?

I know! With a cmdlet. Get-ChildItem?

Exactly. But Get-ChildItem returns all items, not just directories.

If you look at Get-ChildItem , you'll see that directories have a specific mode: "d - r- - - ".

We're now going to see how to select only d-r--- mode items. To do this, we'll use another cmdlet: Where-Object .

This cmdlet specializes in selecting items from an object based on criteria.

Now you're probably wondering how to tell Where-Object how to filter the data from Get-ChildItem. That's where pipelines come in!

Pipelines?!

Yes — pipelines. Imagine you have a cmdlet that returns objects. You want those output objects to become the input for the next cmdlet. That's exactly what pipelines are for!

Our command to select folders becomes:

Get-ChildItem | Where-Object {$_.Mode -eq "d-r---"}

Let me break that down. The first cmdlet, Get-ChildItem, outputs a list of objects representing the items in the folder.

Next, we add the pipe character, also called the vertical bar.

The "|" (pipe) is obtained by pressing the Alt Gr key + the 6 key (above the T).

Then we use a second cmdlet, Where-Object, which performs the filtering. To provide filtering criteria, we open a script block with curly braces and write $_.Mode -eq "d-r---". Here's what that means:

  • $_.Mode: The $_ variable (or $PSItem) is a special PowerShell variable that holds the current object in the pipeline. For each object returned by the first cmdlet, $_ holds the object currently being processed. The ".Mode" part selects only the Mode property of that object.

  • The -eq is the comparison operator for equality (-equal). You will discover others in the chapter "Add logic to meet real needs" in Part 2.

  • "d-r---" is the string that matches the mode used for folders.

Not every pipeline needs to use the special variable $_.

Let's try it together! Let's go back to our example with Get-WmiObject . You may recall that on its own, a Get-WmiObject -class win32_operatingsystem n'affiche pas beaucoup d'informations.

By using the pipe with the Select-Object cmdlet, we can display a lot more information.

Select-Object is a cmdlet that lets you select the properties of an object or a set of objects.

Go ahead! Test cmdlet chaining with Get-WmiObject et Select-Object by entering the following cmdlets in your PowerShell prompt:

Get-WmiObject -Class Win32_OperatingSystem | Select-Object -Property *

Now that's much more useful. But what if I only want the computer name?

In that case, you just need to slightly modify your filtering option to:

Get-WmiObject -Class Win32_OperatingSystem | Select-Object -Property PSComputerName

There you go! You've successfully extracted system information by chaining cmdlets. Congratulations!

Variables can do more than just hold values. Let's look at a few transformations you can apply to strings.

If you remember the video from chapter 2 of this part, we told you that PowerShell is built on object-oriented programming (OOP). You can see one of those aspects in string manipulation.

Every string has methods — built-in functions that let you perform operations on it.

You're about to try a few of them!

At your command prompt, type this: $welcomeMessage.ToUpper() and run the command.

See that? The text appears in uppercase.

Let's break down what we just wrote.

The first thing you typed was your variable. Then you added a period. That period tells PowerShell which object you want to work with.

Finally, you added the function "ToUpper()", which turns a string into uppercase. You can recognize a function or method by the parentheses, which can hold parameters.

When you apply a transformation like this, remember that it is not saved. Methods such as ToUpper() do not change the variable directly. To change the stored value, you need to reassign it by doing $welcomeMessage = $welcomeMessage.ToUpper()

Now try ToLower()! Unlike ToUpper(), this one converts the string to lowercase.

One more string method is worth knowing.

This method is called "Substring()". Think of it as a pair of scissors for strings. Its syntax is as follows:

$welcomeMessage.Substring(starting character position, number of characters)

Or, if you want to keep the rest of the string starting at a given character:

$welcomeMessage.Substring(starting character position)

That all sounds pretty complicated…

Don't worry. It is much simpler than it looks. Try it with the string stored in your variable $welcomeMessage.

$welcomeMessage.Substring(1)

or

$welcomeMessage.Substring(2)

Or still

$welcomeMessage.Substring(0,2)

See? Here I used several string methods to:

  • Convert text to uppercase → .ToUpper()

  • Convert it to lowercase → .ToLower()

  • Get the string starting from the 2nd character → .Substring(1) (note: the first character in a string has index 0).

  • Get the string starting from the 3rd character → .Substring(2)

  • Get the string between the 1st and 3rd characters → .Substring(0,2)

So if I understand correctly, to get the first letter of my first name, I could use $myFirstName.Substring(0,1), right?

Exactly!

Use the if{}…else{} block

For conditional structures, this block is the most intuitive and commonly used.

It can be translated as:

If… Otherwise

Its syntax is simple:

if (<test==true>;) {
    instructions
}
else {

The if{}…else{} block is a conditional structure, but it's not the only one. In other scripting languages, such as Bash available on Linux, it is also called test.

[int]$enteredNumber = Read-Host "Enter a number"

{

Use the for{} loop

The for loop can be translated as "Do N times".

This is the loop we'll mainly use when we know the number of iterations.

This is an iterative loop. An iteration is the repetition of a calculation, an action, or a reasoning step.

Once again, we can go back to our "I like repetitive comedy!" example.

for ($iterator=0; $iterator -lt 3; $iterator++)
{
    Write-Host "I like repetitive comedy!"
}

The for loop in PowerShell uses 3 pieces of information at declaration time:

  • an iterator and its starting value. In our case, that is the variable $iterator = 0;

  • the test to check when to exit the loop. Here, it checks whether the iterator value is less than 3 using the test $iterator -lt 3;

  • the increment value. We choose to increment by 1 each time with our variable $iterator. But we could also increase by 5 or by 10.

So what's the point of a for loop? We could just as well have done the same thing with a while loop, right?

Well, in that case… absolutely! There's no requirement to use one loop over the other.

Using a for loop when we know the number of iterations is a convention, not a requirement.

Use the foreach{} loop

The last loop we'll cover in this chapter is the foreach loop, which translates as "For each".

This loop is used exclusively with what we call a collection. A collection can be an array, lines from a text file, the result of a cmdlet, or a CSV (Comma-Separated Values) file, for example.

The foreach loop has no test to manage directly. But the logic would be: for ($iterator = 0; $iterator -lt $collection.length; $iterator++){$item=$collection[$iterator]}

The syntax is as follows:

foreach ($<item> in $<collection>;)
{
    #instructions
}

Let's take an example with an array of first names:

$firstNames = "Erwann", "Yann", "Gwendall"

We can loop through the list of first names, assigning a variable in each iteration to hold the current item from our array.

This gives the following code:

$firstNames = "Erwann", "Yann", "Gwendall"
foreach ($firstName in $firstNames)
{
    Write-Host "Hello $firstName"
}

Paired with a CSV file, you get a great foundation for automating bulk actions.

Finally, the foreach loop can also use another syntax. We can pass the collection directly through a pipeline and use Foreach-Object afterwards.

$firstNames = "Erwann", "Yann", "Gwendall"
$firstNames |ForEach-Object {"Hello "+$_}

Both syntaxes are equivalent. Use whichever one you're most comfortable with.

What if we go back to our previous example? Ideally, instead of writing x different tests, we'd want to say something like:

"Try to create my folder — if it works, great, if not, tell me why."

And for that, PowerShell, like many other development languages, has blocks designed for this purpose: the try{}, catch{}, and finally{} blocks.

These blocks can be described as follows:

"Try to do what's in the try{} block, and if it doesn't work, catch the error and do what's in the catch{} block."

The finally{} would be the equivalent of:

"And finally, no matter what happens, you do this"

You cannot use a try{} block without a catch{} block after it. That would make no sense and would trigger a runtime error.

Test the following block and then let's analyze the syntax of our try{} and catch{} blocks.

try {
    Get-Something
    }
catch {
    Write-Host "An error occurred: $_"
}

If everything goes as expected, you should get a message telling you that an error occurred, followed by the error itself.

You should notice that you don't get a red error as usual. That's normal, because by using try{}...catch{} blocks, you catch the error before it appears.

Try it: if you use only Get-Something , you'll get a nice red error.

From now on, try to put your cmdlets inside try{}...catch{} blocks as often as possible. The finally{} block is less commonly used in practice.

PowerShell only goes into the catch{} block if the error is considered blocking. However, by default, PowerShell is configured to continue execution even if a non-blocking error appears.

Did you notice that in my catch{} block, I'm using the special pipeline variable, "$_"?

In fact, by doing this, I am calling another special variable we haven't talked about yet: $Error. This special variable has one function: storing errors. By using $_, I'm actually accessing this $Error variable, but only asking it to give me a summary of the error, not the full error.

Try displaying $Error instead of $_ — you'll see there are a few extra details.

As I mentioned, the catch{} block will only run if an error is considered blocking.

Likewise, for specific reasons, you might want to completely ignore a non-blocking error. Again, PowerShell handles this well.

A parameter exists to control and "force" an error to be blocking or not, and defines how PowerShell should handle it.

This parameter is -ErrorAction .

ErrorAction has several possible values:

  • Continue: Produces an error message and continues execution.

  • Ignore: Does not produce an error message, continues execution. No error is added to the $Error variable.

  • Inquire: Produces an error message and asks the user for confirmation to stop or continue execution.

  • SilentlyContinue: Does not produce an error message, continues execution. Errors are added to the $Error variable.

  • Stop: This option forces the generation of an error and stops execution of the try{} block.

If you want to use the same ErrorAction value throughout the script, you can define the variable $ErrorActionPreference at the beginning of the script. For example: $ErrorActionPreference = "Stop" will force every error to be treated as blocking.
