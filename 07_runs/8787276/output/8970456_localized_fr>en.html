Configure IPsec

Configure the IPsec VPN

You will configure Phase 1 of the IPsec VPN, which establishes a secure channel for negotiating the tunnel.

OC-SITE-A(config)#crypto isakmp policy 1

OC-SITE-A(config-isakmp)#authentication pre-share

OC-SITE-A(config-isakmp)#encryption aes 256

OC-SITE-A(config-isakmp)#hash sha

OC-SITE-A(config-isakmp)#group 5

OC-SITE-A(config-isakmp)#lifetime 21600

 

Let me explain what each of the commands above does.

  • crypto isakmp policy 1: creates an ISAKMP policy that uses IKE for negotiation.

  • authentication pre-share: sets up pre-shared key authentication.

  • encryption aes 256: defines the encryption algorithm used to secure exchanges.

  • group 5: defines Diffie-Hellman group 5, which uses 1536 bits. This is the maximum value in Packet Tracer. On a real router or firewall, you should preferably use group 15 with 3072 bits.

  • hash sha: defines the hashing algorithm used to ensure data integrity. As with Diffie-Hellman, Packet Tracer does not let you define a hash value. On real equipment, SHA512 is recommended.

  • lifetime 21600: defines the lifetime of the ISAKMP security association. When this time expires, a new SA must be established. Note that on real equipment, if the peer is lost, you can enable DPD (Dead Peer Detection). This mechanism is optional in IKEv1 and built into IKEv2. It quickly detects a lost peer and triggers renegotiation or deletion of the SA before the timer expires.

Here is a brief comparison between IKEv1 and IKEv2: DPD in IKEv1 uses R-U-THERE and R-U-THERE-ACK messages, sent periodically if no traffic is detected in the tunnel. IKEv2 uses an empty message to test whether the peer is responsive. If the peer responds, the SA is active. If not, it is considered invalid. DPD with IKEv2 reduces failure detection time, maintains security by removing obsolete SAs, and supports fast renegotiation.

The configuration is exactly the same—and must be exactly the same—on OC-SITE-B:

OC-SITE-B(config)#crypto isakmp policy 1

OC-SITE-B(config-isakmp)#authentication pre-share

OC-SITE-B(config-isakmp)#encryption aes 256

OC-SITE-B(config-isakmp)#hash sha

OC-SITE-B(config-isakmp)#group 5

OC-SITE-B(config-isakmp)#lifetime 21600

The next step in Phase 1 is to define the pre-shared key with the peer: 

OC-SITE-A(config)#crypto isakmp key OpenClassR00m$IP5ecL@b address 59.62.1.1

OC-SITE-B(config)#crypto isakmp key OpenClassR00m$IP5ecL@b address 62.59.1.1

As you can see, the same key is entered on both routers. In a real-world situation, you might prefer to use certificates for authentication. The advantage is that they rely on asymmetric key pairs, which are harder to compromise, easy to revoke using a CRL, well suited to multisite environments, and provide strong authentication by verifying the certificate signature and validity. We cannot use them here in this simulation.

Before configuring Phase 2, you will configure an ACL on OC-SITE-A and OC-SITE-B to define the traffic that must pass through the tunnel: 

OC-SITE-A(config)#ip access-list extended VPN

OC-SITE-A(config-ext-nacl)#permit ip 192.168.1.0 0.0.0.255 192.168.2.0 0.0.0.255


OC-SITE-B(config)#ip access-list extended VPN

OC-SITE-B(config-ext-nacl)#permit ip 192.168.2.0 0.0.0.255 192.168.1.0 0.0.0.255

Once the VPN ACLs have been created, you will configure Phase 2. Start by creating the transform set, which calls the ESP protocol. ESP encrypts and ensures the integrity of the data that passes through the IPsec tunnel: 

OC-SITE-A(config)#crypto ipsec transform-set OCSET esp-aes 256 esp-sha-hmac


OC-SITE-B(config)#crypto ipsec transform-set OCSET esp-aes 256 esp-sha-hmac

In the command above, which is the same on OC-SITE-A and OC-SITE-B, you create a transform set named OCSET that uses ESP with AES 256 for encryption and SHA for integrity.

Finally, you will create a crypto map, which is an object that includes: 

  • The transform set

  • The ACL used to identify the traffic that must pass through the VPN

  • The Phase 2 security association lifetime

  • The peer address

Before creating this crypto map, let’s review the SA: 

  • In Phase 1, a security association (SA) is established with a lifetime of 21600 seconds (6 hours). When it expires, a new Diffie-Hellman (DH) key pair is generated, reducing the risk of long-term key compromise.

  • In Phase 2, the keys used to encrypt and authenticate data are derived from the key established in Phase 1. A full Diffie-Hellman exchange is not performed each time a Phase 2 SA expires, unless PFS (Perfect Forward Secrecy) is enabled. In that case, a new key is generated, which strengthens security: even if one key is compromised, the other sessions remain protected. The lifetime of a Phase 2 SA is generally configured at 3600 seconds (1 hour).

Each IPsec peer generates a private key and a public key. The peers exchange their public keys, then each one locally calculates a shared key using its own private key and the public key it received. This gives both peers the same shared key without ever transmitting it over the network. This mechanism, known as Diffie-Hellman, is the central element of the IPsec protocol. 

Do not confuse the pre-shared key (PSK) used for authentication between peers, which can be replaced by certificates, with the Diffie-Hellman shared key, also known as the "shared secret." These are two completely different mechanisms.

Here is the crypto map configuration on OC-SITE-A and OC-SITE-B:  

OC-SITE-A(config)#crypto map OCMAP 10 ipsec-isakmp

OC-SITE-A(config-crypto-map)#set transform-set OCSET

OC-SITE-A(config-crypto-map)#set security-association lifetime seconds 3600

OC-SITE-A(config-crypto-map)#set peer 59.62.1.1

OC-SITE-A(config-crypto-map)#match address VPN

OC-SITE-A(config-crypto-map)#exit


OC-SITE-B(config)#crypto map OCMAP 10 ipsec-isakmp

OC-SITE-B(config-crypto-map)#set transform-set OCSET

OC-SITE-B(config-crypto-map)#set security-association lifetime seconds 3600

OC-SITE-B(config-crypto-map)#set peer 62.59.1.1

OC-SITE-B(config-crypto-map)#match address VPN

OC-SITE-B(config-crypto-map)#exit

After the name given to the crypto map, OCMAP in this case, you will see the number 10. This is a sequence number. You can apply only one crypto map per interface, so for each different VPN, if you have several—and you will later in this course—you must use a different sequence number for each VPN.

The other commands are fairly self-explanatory: you set the Phase 2 SA lifetime to 3600 seconds, call the transform set OCSET, define the remote peer’s IP address, and call the ACL used to identify the traffic that can pass through the tunnel.

Now you just need to apply this crypto map to the interface where your public IP address is configured: 

OC-SITE-A(config)#int gig 0/1

OC-SITE-A(config-if)#crypto map OCMAP


OC-SITE-B(config)#int gig 0/1

OC-SITE-B(config-if)#crypto map OCMAP

A log message confirms that the crypto map has been activated.

*Jan 3 07:16:26.785: %CRYPTO-6-ISAKMP_ON_OFF: ISAKMP is ON

Now test the configuration: run a continuous ping between each PC:

Here is the basic configuration you will set up on OC-SITE-C: 

OC-SITE-C(config)#int gig 0/1/0

OC-SITE-C(config-if)#ip address 80.59.62.1 255.255.255.252

OC-SITE-C(config-if)#no shut

OC-SITE-C(config-if)#ip nat outside

OC-SITE-C(config-if)#exit

OC-SITE-C(config)#int gig 0/0

OC-SITE-C(config-if)#ip address 192.168.3.254 255.255.255.0

OC-SITE-C(config-if)#ip nat inside

OC-SITE-C(config-if)#no shut

OC-SITE-C(config-if)#exit


OC-SITE-C(config)#ip route 0.0.0.0 0.0.0.0 80.59.62.2


OC-SITE-C(config)#ip access-list extended NAT

OC-SITE-C(config-ext-nacl)#deny ip 192.168.3.0 0.0.0.255 192.168.1.0 0.0.0.255

OC-SITE-C(config-ext-nacl)#deny ip 192.168.3.0 0.0.0.255 192.168.2.0 0.0.0.255

OC-SITE-C(config-ext-nacl)#permit ip 192.168.3.0 0.0.0.255 any


OC-SITE-C(config)#ip nat inside source list NAT interface gig 0/1/0 overload

Notice that in the NAT ACL, traffic from LAN 192.168.3.0/24 to 192.168.1.0/24 and 192.168.2.0/24 is denied. This is normal because these traffic flows must pass through the VPNs.

You will also need to update the NAT ACL on OC-SITE-A and OC-SITE-B: 

OC-SITE-A(config)#ip access-list extended NAT

OC-SITE-A(config-ext-nacl)#15 deny ip 192.168.1.0 0.0.0.255 192.168.3.0 0.0.0.255


OC-SITE-B(config)#ip access-list extended NAT

OC-SITE-B(config-ext-nacl)#15 deny ip 192.168.2.0 0.0.0.255 192.168.3.0 0.0.0.255

You probably noticed the "15" before the ACL entry that was added. This 15 is the sequence number. By default, sequence numbers increase by 10. In other words, each time you add a line to your ACL, it receives a sequence number incremented by 10: 10, 20, 30, 40, and so on. 

If you want to add a deny sequence before the sequence that allows all remaining traffic to be translated, choose a sequence number lower than the one that permits all remaining traffic to be translated. This gives you the following result when you display the ACL: 

OC-SITE-A#show access-lists NAT

Extended IP access list NAT

deny ip 192.168.1.0 0.0.0.255 192.168.2.0 0.0.0.255

deny ip 192.168.1.0 0.0.0.255 192.168.3.0 0.0.0.255

permit ip 192.168.1.0 0.0.0.255 any


OC-SITE-B#show access-lists NAT

Extended IP access list NAT

deny ip 192.168.2.0 0.0.0.255 192.168.1.0 0.0.0.255

deny ip 192.168.2.0 0.0.0.255 192.168.3.0 0.0.0.255

permit ip 192.168.2.0 0.0.0.255 any

As you can see above, the new deny sequence is correctly positioned in the ACL.

Next, create the ACLs dedicated to this second VPN: 

OC-SITE-C(config)#ip access-list extended C-TO-A

OC-SITE-C(config-ext-nacl)#permit ip 192.168.3.0 0.0.0.255 192.168.1.0 0.0.0.255

OC-SITE-C(config-ext-nacl)#exit

OC-SITE-C(config)#ip access-list extended C-TO-B

OC-SITE-C(config-ext-nacl)#permit ip 192.168.3.0 0.0.0.255 192.168.2.0 0.0.0.255

OC-SITE-C(config-ext-nacl)#exit

OC-SITE-A(config)#ip access-list extended A-TO-C

OC-SITE-A(config-ext-nacl)#permit ip 192.168.1.0 0.0.0.255 192.168.3.0 0.0.0.255

OC-SITE-B(config)#ip access-list extended B-TO-C

OC-SITE-B(config-ext-nacl)#permit ip 192.168.2.0 0.0.0.255 192.168.3.0 0.0.0.255

Enable the SECURITY module on OC-SITE-C: 

 

OC-SITE-C(config)#license boot module c1900 technology-package securityk9

 Next, configure the first part of Phase 1 on OC-SITE-C in the same way as on OC-SITE-A and OC-SITE-B: 

OC-SITE-C(config)#crypto isakmp policy 1

OC-SITE-C(config-isakmp)#authentication pre-share

OC-SITE-C(config-isakmp)#encryption aes 256

OC-SITE-C(config-isakmp)#hash sha

OC-SITE-C(config-isakmp)#group 5

OC-SITE-C(config-isakmp)#lifetime 21600

The next step is to create a pre-shared key between OC-SITE-C and OC-SITE-A, and another pre-shared key between OC-SITE-C and OC-SITE-B. You will also configure the OC-SITE-C to OC-SITE-A key on OC-SITE-A, and the OC-SITE-C to OC-SITE-B key on OC-SITE-B: 

OC-SITE-C(config)#crypto isakmp key OpenClassR00m$IP5ecL@bCtoA address 62.59.1.1

OC-SITE-C(config)#crypto isakmp key OpenClassR00m$IP5ecL@bCtoB address 59.62.1.1


OC-SITE-A(config)#crypto isakmp key OpenClassR00m$IP5ecL@bCtoA address 80.59.62.1


OC-SITE-B(config)#crypto isakmp key OpenClassR00m$IP5ecL@bCtoB address 80.59.62.1

 

You then need to configure the transform set on OC-SITE-C: 

OC-SITE-C(config)#crypto ipsec transform-set OCSET esp-aes 256 esp-sha-hmac

 Once this step is complete, you will need to create a crypto map sequence on OC-SITE-A and OC-SITE-B to establish the VPN with OC-SITE-C: 

OC-SITE-A(config)#crypto map OCMAP 20 ipsec-isakmp

OC-SITE-A(config-crypto-map)#set peer 80.59.62.1

OC-SITE-A(config-crypto-map)#set transform-set OCSET

OC-SITE-B(config-crypto-map)#match address B-TO-C

OC-SITE-B(config-crypto-map)#set peer 80.59.62.1

OC-SITE-B(config-crypto-map)#set security-association lifetime seconds 3600

OC-SITE-C(config)#crypto map OCMAP 20 ipsec-isakmp

Let's Recap!

  • Phase 1 of the IPsec VPN establishes a secure channel between two sites using a common ISAKMP policy on both routers: pre-shared key authentication, AES 256 encryption, SHA hash, and DH group 5.

  • A dedicated ACL defines the traffic flows that must pass through the tunnel, while a transform set configures the use of ESP for data encryption and integrity (AES + SHA).

  • The crypto map brings together all the elements required for the IPsec tunnel: transform set, ACL, peer, and SA lifetime. It is applied to the router’s public interface. Each tunnel uses a different sequence number.

  • The Diffie-Hellman protocol generates a shared key without exchanging it, providing a high level of security. The PFS option further strengthens this security by forcing a new DH exchange at each key renewal.

  • Adding a third site, OC-SITE-C, requires creating new ACLs, crypto maps, and shared keys, as well as updating the existing NAT ACLs. This allows each router to manage several IPsec tunnels in parallel.
